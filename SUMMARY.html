Corpus Analysis — The Hindi Problem

Our own measurement revealed the core bottleneck:

32.9%

With the byte tokenizer, block_size=128 covers only ~42 Hindi characters of context. With BPE-4096, the same window covers ~384 tokens of compressed bilingual content — a 9× improvement in effective Hindi context.

What Did NOT Help

  • SwiGLU activation: Param-matched SwiGLU (narrower MLP) gave no gain over GELU — activation benefit cancelled by reduced MLP width.
  • BPE-8192: Diminishing compression returns; rare tokens get too few gradient updates in 2000 steps.
  • 5 transformer layers: Deeper model underfit more — gradient signal dissipated across too many residual connections for this step budget.
  • Dropout > 0: No benefit; corpus is large enough relative to model size that regularization is unnecessary.

Reproduce

# 1. Train BPE Tokenizer (~50s)
python train_tokenizer.py --data data/train_corpus.txt --merges 3840

# 2. Train Model (~8-10 min on CPU)
python train.py \
  --data data/train_corpus.txt --steps 2000 --out ckpt.pt \
  --optimizer muon --muon_lr 0.02 --lr 3e-3 \
  --n_embd 176 --n_layer 3 --n_head 8 --block_size 192 \
  --batch 32 --schedule wsd --warmup 100 --decay_start 1600 \
  --ema --ema_decay 0.995 --ema_start 1500

# 3. Evaluate
python evaluate.py --checkpoint ckpt.pt --text_file data/dev_eval.txt

All training on CPU · No pretrained weights · No external data · Pure PyTorch
